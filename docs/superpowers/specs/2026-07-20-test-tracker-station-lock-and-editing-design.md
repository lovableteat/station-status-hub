# 測試追蹤站別鎖定與機台編輯設計

## 目標

改善「L10 測試追蹤」表格的三項操作：

1. 點擊某機台的某個站別時，只顯示該機台、該站別的測項，且不能切換到其他站別。
2. 編輯多個測項時，儲存其中一項所觸發的資料刷新不得覆蓋其他尚未儲存的測項草稿。
3. 點擊機台 ID 時，直接開啟該機台的系統資料編輯器。

## 既有問題與原因

### 站別點擊未帶入站別

目前站別格只把 `systemId` 傳給上層，因此進度面板只能沿用舊的站別選取或預設第一站，無法知道使用者實際點擊哪一站。

### 測項草稿被刷新

進度資料儲存後會觸發重新載入或即時更新。進度面板原本會用伺服器資料重建整份草稿，因此另一個已修改但尚未儲存的測項會被伺服器舊值覆蓋。現有修正已開始以逐測項 dirty 狀態合併伺服器資料，本次需保留並補足回歸驗證。

### 機台 ID 與進度共用入口

機台 ID 目前不是系統資料編輯的主要入口，使用者需要再進入操作選單。機台 ID 應直接對應系統資料，站別格與「編輯進度」則對應測試進度。

## 採用方案

沿用現有 `SystemProgressSheet`，新增可選的鎖定站別參數，不建立第二套進度元件。

### 選取狀態

`TestTracker` 同時管理：

- 目前開啟進度面板的機台 ID。
- 由站別格點擊所指定的鎖定站別 ID；一般「編輯進度」入口則為空值。
- 目前開啟系統資料編輯器的機台 ID。

關閉進度面板時必須一起清除機台與鎖定站別，避免下次開啟沿用舊站別。

### 站別鎖定模式

點擊表格站別格時，`TestProgressTable` 回傳 `systemId` 與 `stationId`。`TestTracker` 以這兩個值開啟 `SystemProgressSheet`。

當面板收到鎖定站別 ID 時：

- 選取站別固定為指定站別。
- 只顯示該站別的進度摘要與測項。
- 不渲染其他站別的切換控制。
- 站別資料不存在時不回退到其他站別，改顯示無可用站別的安全狀態。

從一般「編輯進度」按鈕開啟時不帶鎖定站別，維持既有可切換全部站別的行為。

### 草稿隔離

草稿以測項 ID 為單位標記 dirty。伺服器資料更新時：

- dirty 測項保留本地草稿。
- 未修改測項接受最新伺服器值。
- 單項儲存成功後只清除該測項的 dirty 標記。
- 切換到不同機台或關閉面板時重設草稿內容與 dirty 集合。

儲存第二個測項不得使第一個尚未儲存的「已完成」狀態、百分比或備註消失。

### 機台 ID 編輯

桌面表格與行動版卡片的機台 ID／名稱區域直接開啟受控的 `SystemEditDialog`。站別格、整體進度與主要「編輯進度」按鈕仍開啟進度面板，兩種用途不混用。

## 錯誤與邊界處理

- 鎖定站別不在目前流程版本時，不自動跳到第一站。
- 系統資料編輯器關閉時清除其機台 ID。
- 儲存失敗時保留本地草稿與 dirty 標記，讓使用者可重試。
- 儲存成功後再接受該測項的伺服器更新。
- 既有未提交的資料中心與機台編輯修改不得被覆蓋或回復。

## 測試與驗證

先建立會失敗的測試，再修改正式程式：

1. 元件來源測試：站別格點擊必須同時傳出機台 ID 與站別 ID。
2. 元件來源測試：鎖定站別模式不顯示站別切換控制，且只以鎖定站別篩選測項。
3. 草稿單元測試：儲存／刷新其中一項後，其他 dirty 測項維持本地值。
4. 元件來源測試：機台 ID 點擊開啟 `SystemEditDialog`。
5. 執行相關 Node 測試、完整建置與必要的 lint 檢查。
6. 在瀏覽器驗證桌面與行動版：站別鎖定、跨測項草稿保留、機台 ID 編輯，以及控制台無相關錯誤。

## 完成標準

- 任一站別格都開啟正確機台與正確站別。
- 鎖定模式只顯示被點擊站別且不能切換。
- 儲存第二項不會刷新第一項未存草稿。
- 點擊機台 ID 直接進入系統資料編輯。
- 相關測試與建置通過，瀏覽器目標流程驗證完成。
